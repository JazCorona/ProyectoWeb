Tecnologías Web API

Las APIs web son interfaces que permiten la comunicación entre diferentes sistemas de software a través de la web, formando la columna vertebral de las aplicaciones modernas.

HTTP (HyperText Transfer Protocol)

HTTP es el protocolo fundamental para la transferencia de datos en la web. Define cómo se formatean y transmiten los mensajes, y cómo deben responder los servidores y navegadores a diferentes comandos.

Características principales:

  • Protocolo sin estado (stateless)
  • Basado en el modelo cliente-servidor
  • Extensible mediante cabeceras
  • Soporta diferentes tipos de contenido (MIME)
  • Versiones principales: HTTP/1.0, HTTP/1.1, HTTP/2, HTTP/3

Métodos HTTP principales:

GET

Solicita una representación del recurso especificado. Solo debe recuperar datos.

POST

Envía datos para ser procesados por el recurso identificado. A menudo causa cambio de estado.

PUT

Actualiza el recurso especificado con los datos proporcionados. Es idempotente.

DELETE

Elimina el recurso especificado.

PATCH

Aplica modificaciones parciales a un recurso.

Códigos de estado HTTP:

2xx - Éxito

200 OK, 201 Created, 204 No Content

3xx - Redirección

301 Moved, 302 Found, 304 Not Modified

4xx - Error cliente

400 Bad Request, 401 Unauthorized, 404 Not Found

5xx - Error servidor

500 Internal Server Error, 503 Service Unavailable

Ejemplo de petición HTTP:

GET /api/users HTTP/1.1 Host: api.example.com Accept: application/json Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...

Ejemplo de respuesta HTTP:

HTTP/1.1 200 OK Content-Type: application/json Cache-Control: max-age=3600 { "users": [ {"id": 1, "name": "Juan Pérez", "email": "juan@example.com"}, {"id": 2, "name": "Ana García", "email": "ana@example.com"} ], "total": 2, "page": 1 }

SOAP (Simple Object Access Protocol)

SOAP es un protocolo estándar que define cómo dos objetos en diferentes procesos pueden comunicarse por medio del intercambio de datos XML. Es un protocolo más rígido y formal que REST.

Características principales:

  • Independiente del protocolo de transporte (HTTP, SMTP, TCP, etc.)
  • Fuertemente tipado mediante WSDL (Web Services Description Language)
  • Soporta operaciones complejas y transacciones
  • Incluye mecanismos de seguridad integrados (WS-Security)
  • Manejo de errores estandarizado

Estructura de un mensaje SOAP:

Envelope (Sobre)

Elemento raíz que identifica el documento XML como un mensaje SOAP.

Header (Cabecera)

Contiene información de cabecera como autenticación, transacciones, etc. Es opcional.

Body (Cuerpo)

Contiene la información de llamada y respuesta. Es obligatorio.

Fault (Fallo)

Información de errores y estados. Es opcional y va dentro del Body.

Ventajas y desventajas:

Ventajas
  • Independiente del lenguaje
  • Independiente de la plataforma
  • Altamente extensible
  • Contrato formal (WSDL)
  • Seguridad integrada
Desventajas
  • Mayor sobrecarga
  • Más complejo de implementar
  • Mensajes más pesados
  • Curva de aprendizaje más pronunciada
  • Menos eficiente para dispositivos móviles

Ejemplo de mensaje SOAP:

<?xml version="1.0"?> <soap:Envelope xmlns:soap="http://www.w3.org/2003/05/soap-envelope"> <soap:Header> <auth:Credentials xmlns:auth="http://example.org/auth"> <auth:Username>user123</auth:Username> <auth:Password>pass456</auth:Password> </auth:Credentials> </soap:Header> <soap:Body> <m:GetUser xmlns:m="http://example.org/users"> <m:UserId>12345</m:UserId> </m:GetUser> </soap:Body> </soap:Envelope>

Ejemplo de WSDL (Web Services Description Language):

<?xml version="1.0"?> <definitions name="UserService" targetNamespace="http://example.org/users" xmlns="http://schemas.xmlsoap.org/wsdl/" xmlns:soap="http://schemas.xmlsoap.org/wsdl/soap/" xmlns:tns="http://example.org/users" xmlns:xsd="http://www.w3.org/2001/XMLSchema"> <message name="GetUserRequest"> <part name="userId" type="xsd:string"/> </message> <message name="GetUserResponse"> <part name="user" type="tns:User"/> </message> <portType name="UserPortType"> <operation name="GetUser"> <input message="tns:GetUserRequest"/> <output message="tns:GetUserResponse"/> </operation> </portType> <binding name="UserBinding" type="tns:UserPortType"> <soap:binding style="rpc" transport="http://schemas.xmlsoap.org/soap/http"/> <operation name="GetUser"> <soap:operation soapAction="http://example.org/GetUser"/> <input> <soap:body use="encoded" namespace="http://example.org/users" encodingStyle="http://schemas.xmlsoap.org/soap/encoding/"/> </input> <output> <soap:body use="encoded" namespace="http://example.org/users" encodingStyle="http://schemas.xmlsoap.org/soap/encoding/"/> </output> </operation> </binding> <service name="UserService"> <port name="UserPort" binding="tns:UserBinding"> <soap:address location="http://example.org/users"/> </port> </service> </definitions>

REST (Representational State Transfer)

REST es un estilo arquitectónico para sistemas distribuidos, no un protocolo. Define un conjunto de restricciones para crear servicios web escalables. Fue definido por Roy Fielding en su tesis doctoral en 2000.

Principios REST:

  • Cliente-Servidor:

    Separación de responsabilidades entre la interfaz de usuario y el almacenamiento de datos.

  • Sin estado (Stateless):

    Cada petición del cliente al servidor debe contener toda la información necesaria para entender y completar la petición.

  • Cacheable:

    Las respuestas deben definir explícitamente si son cacheables o no para mejorar la eficiencia.

  • Interfaz uniforme:

    Simplifica la arquitectura y mejora la visibilidad de las interacciones.

  • Sistema en capas:

    Componentes jerárquicos que no pueden "ver" más allá de su capa inmediata.

  • Código bajo demanda (opcional):

    Permite extender la funcionalidad del cliente descargando y ejecutando código.

Recursos y URIs:

En REST, todo es un recurso que se identifica mediante URIs. Las operaciones sobre estos recursos se realizan mediante métodos HTTP estándar.

Ejemplos de URIs RESTful:
  • GET /api/users Obtener todos los usuarios
  • GET /api/users/123 Obtener usuario con ID 123
  • POST /api/users Crear un nuevo usuario
  • PUT /api/users/123 Actualizar usuario con ID 123
  • DELETE /api/users/123 Eliminar usuario con ID 123

Formatos de representación:

JSON

Formato ligero y fácil de parsear, preferido en la mayoría de las APIs modernas.

{ "id": 123, "name": "Juan Pérez", "email": "juan@example.com" }
XML

Más estructurado y extensible, pero más verboso que JSON.

<user> <id>123</id> <name>Juan Pérez</name> <email>juan@example.com</email> </user>

Niveles de madurez REST (Modelo de Richardson):

Nivel 0

Un único URI, un único método HTTP (generalmente POST). Ejemplo: SOAP sobre HTTP.

Nivel 1

Múltiples URIs (recursos), un único método HTTP. Ejemplo: servicios con múltiples endpoints pero solo POST.

Nivel 2

Múltiples URIs, uso de métodos HTTP. La mayoría de las APIs "REST" están en este nivel.

Nivel 3

HATEOAS (Hypermedia as the Engine of Application State). La API proporciona enlaces para navegar entre recursos.

Ejemplo de respuesta REST con HATEOAS (Nivel 3):

{ "id": 123, "name": "Juan Pérez", "email": "juan@example.com", "role": "admin", "created_at": "2023-05-15T14:30:00Z", "_links": { "self": { "href": "/api/users/123" }, "profile": { "href": "/api/users/123/profile" }, "orders": { "href": "/api/users/123/orders" } } }

Ventajas de REST:

Rendimiento

Mejor rendimiento y escalabilidad gracias al uso de caché y la naturaleza sin estado.

Simplicidad

Más fácil de entender y usar que alternativas como SOAP. Curva de aprendizaje menor.

Compatibilidad

Ideal para dispositivos móviles y aplicaciones web modernas por su menor sobrecarga.

JWT (JSON Web Tokens)

JWT es un estándar abierto (RFC 7519) que define una forma compacta y autónoma para transmitir información de forma segura entre partes como un objeto JSON. Es ampliamente utilizado para autenticación y autorización en aplicaciones web modernas.

Características principales:

  • Compacto: Puede ser enviado a través de URL, POST o en cabeceras HTTP
  • Autónomo: Contiene toda la información necesaria sobre el usuario
  • Seguro: Firmado digitalmente usando algoritmos como HMAC o RSA
  • Stateless: No requiere almacenar sesiones en el servidor
  • Extensible: Puede contener claims personalizados según las necesidades
Casos de uso comunes:
  • Autenticación de usuarios
  • Intercambio seguro de información
  • Autorización entre servicios
  • Single Sign-On (SSO)
  • Tokens de acceso para APIs

Estructura de un JWT:

Header
Payload
Signature

xxxxx.yyyyy.zzzzz

Header

Contiene el tipo de token y el algoritmo de firma.

{ "alg": "HS256", "typ": "JWT" }
Payload

Contiene las declaraciones (claims) sobre la entidad y datos adicionales.

{ "sub": "1234567890", "name": "John Doe", "admin": true, "iat": 1516239022, "exp": 1516242622 }
Signature

Verifica que el mensaje no ha sido alterado en el camino.

HMACSHA256( base64UrlEncode(header) + "." + base64UrlEncode(payload), secret )

Flujo de autenticación con JWT:

  1. Inicio de sesión: El usuario envía credenciales (usuario/contraseña) al servidor
  2. Verificación: El servidor verifica las credenciales contra la base de datos
  3. Generación de token: El servidor crea un JWT con la información del usuario y lo firma
  4. Respuesta: El servidor envía el JWT al cliente
  5. Almacenamiento: El cliente guarda el token (localStorage, cookies, etc.)
  6. Peticiones autenticadas: El cliente incluye el JWT en las cabeceras de las peticiones
  7. Validación: El servidor verifica la firma del JWT y procesa la petición

Consideraciones de seguridad:

  • No almacenar información sensible en el payload (es decodificable)
  • Usar HTTPS para transmitir los tokens
  • Establecer tiempos de expiración cortos
  • Implementar lista de tokens revocados si es necesario
  • Usar algoritmos de firma seguros (RS256 sobre HS256 para aplicaciones a gran escala)
  • Validar los claims del token (emisor, audiencia, tiempo de expiración)

Ejemplo de implementación en Node.js:

// Generar un token const jwt = require('jsonwebtoken'); const token = jwt.sign( { userId: user.id, email: user.email }, process.env.JWT_SECRET, { expiresIn: '1h' } ); // Verificar un token try { const decoded = jwt.verify(token, process.env.JWT_SECRET); // Si llega aquí, el token es válido console.log(decoded.userId); } catch (error) { // Token inválido o expirado console.error('Error de autenticación:', error.message); }

Microservicios

Los microservicios son un enfoque arquitectónico que estructura una aplicación como una colección de servicios pequeños, autónomos y con un único propósito.

Principios fundamentales

  • Responsabilidad única

    Cada servicio se enfoca en una única funcionalidad de negocio.

  • Autonomía

    Cada servicio puede ser desarrollado, desplegado y escalado independientemente.

  • Descentralización

    Gestión descentralizada de datos y gobernanza.

  • Aislamiento de fallos

    Los fallos en un servicio no deben afectar a otros servicios.

  • Evolución continua

    Facilita la evolución y actualización de partes específicas del sistema.

Patrones de comunicación

El cliente espera la respuesta del servidor antes de continuar.

Tecnologías:
REST gRPC GraphQL SOAP
Nota: Puede introducir acoplamiento temporal entre servicios.

El cliente no espera una respuesta inmediata y puede continuar con otras tareas.

Tecnologías:
Apache Kafka RabbitMQ AWS SQS Azure Service Bus
Ventaja: Mayor resiliencia y desacoplamiento entre servicios.

Los servicios se comunican mediante eventos que representan cambios de estado.

Patrones:
  • Event Sourcing
  • CQRS (Command Query Responsibility Segregation)
  • Publish-Subscribe
// Ejemplo de evento { "eventType": "OrderCreated", "timestamp": "2023-05-15T14:30:00Z", "data": { "orderId": "ORD-12345", "customerId": "CUST-789", "amount": 99.99 } }

Patrones de resiliencia en microservicios

Circuit Breaker

Reintenta una operación fallida un número limitado de veces con intervalos crecientes.

Intento 2 → falla

Bulkhead Pattern

Ventajas y desafíos de los microservicios

Ventajas

  • Escalabilidad independiente de servicios
  • Equipos autónomos y desarrollo paralelo
  • Facilidad para adoptar nuevas tecnologías
  • Despliegues más rápidos y menos riesgosos
  • Mayor resiliencia del sistema

Desafíos

  • Complejidad en la comunicación entre servicios
  • Gestión de transacciones distribuidas
  • Mayor sobrecarga operativa
  • Complejidad en pruebas de integración
  • Necesidad de monitoreo y trazabilidad avanzados

Proceso de desarrollo de una API

1

Análisis de requisitos

Identifica las necesidades de los usuarios y los casos de uso que la API debe soportar.

  • ¿Quiénes son los consumidores de la API?
  • ¿Qué operaciones necesitan realizar?
  • ¿Qué volumen de tráfico se espera?
  • ¿Qué requisitos de seguridad existen?
2

Diseño de la API

Define los recursos, endpoints, métodos HTTP, formatos de datos y esquemas de respuesta.

Recursos: Usuarios, Productos, Pedidos
Endpoints: /api/v1/users, /api/v1/products, /api/v1/orders
Respuestas: JSON con códigos HTTP apropiados
3

Implementación

Desarrolla la API utilizando el framework y lenguaje elegidos, siguiendo las mejores prácticas.

// Ejemplo en Express.js const express = require('express'); const app = express(); app.use(express.json()); // Middleware de autenticación app.use(authMiddleware); // Endpoint para obtener usuarios app.get('/api/v1/users', async (req, res) => { try { const users = await userService.getAll(); res.json(users); } catch (err) { res.status(500).json({ error: err.message }); } }); app.listen(3000, () => console.log('API running on port 3000'));
4

Pruebas

Realiza pruebas unitarias, de integración y de carga para asegurar la calidad y rendimiento.

Herramientas de prueba:
  • Jest/Mocha para pruebas unitarias
  • Supertest para pruebas de API
  • Postman para pruebas manuales
  • JMeter/k6 para pruebas de carga
Aspectos a probar:
  • Validación de entradas
  • Manejo de errores
  • Autenticación y autorización
  • Rendimiento bajo carga
  • Límites y casos extremos
5

Documentación y despliegue

Documenta la API y despliégala en un entorno de producción con monitoreo.

Documentación: Swagger UI, Redoc
Despliegue: Contenedores Docker, Kubernetes, Serverless
Monitoreo: Logs, métricas, alertas, trazabilidad
INFRAESTRUCTURA

Manejo de Contenedores

Los contenedores proporcionan un entorno consistente para ejecutar aplicaciones, facilitando el desarrollo, pruebas y despliegue.

Docker

Docker es una plataforma para desarrollar, enviar y ejecutar aplicaciones en contenedores. Los contenedores permiten empaquetar una aplicación con todas sus dependencias en una unidad estandarizada.

Conceptos fundamentales

Imágenes

Plantillas de solo lectura con el sistema operativo, aplicación y dependencias.

Contenedores

Instancias ejecutables de una imagen con su propio sistema de archivos, red, etc.

Dockerfile

Archivo de texto con instrucciones para construir una imagen.

Docker Compose

Herramienta para definir y ejecutar aplicaciones multi-contenedor.

Ejemplo de Dockerfile

FROM node:18-alpine WORKDIR /app COPY package*.json ./ RUN npm install COPY . . EXPOSE 3000 CMD ["node", "server.js"]

Comandos esenciales

docker build -t myapp .

Construye una imagen

docker run -p 3000:3000 myapp

Ejecuta un contenedor

docker ps

Lista contenedores en ejecución

docker-compose up

Inicia servicios definidos

Kubernetes

Kubernetes es un sistema de orquestación de contenedores para automatizar el despliegue, escalado y gestión de aplicaciones en contenedores. Fue desarrollado originalmente por Google.

Componentes principales

Pod

Unidad más pequeña en Kubernetes, contiene uno o más contenedores.

Deployment

Gestiona la creación y actualización de réplicas de pods.

Service

Expone pods como un servicio de red, proporcionando IP y DNS estables.

Ingress

Gestiona el acceso externo a los servicios dentro del clúster.

Ejemplo de archivo YAML

apiVersion: apps/v1 kind: Deployment metadata: name: api-deployment spec: replicas: 3 selector: matchLabels: app: api template: metadata: labels: app: api spec: containers: - name: api image: myapi:1.0 ports: - containerPort: 3000 resources: limits: cpu: "0.5" memory: "512Mi" requests: cpu: "0.2" memory: "256Mi"

Comandos esenciales

kubectl apply -f deployment.yaml

Aplica configuración

kubectl get pods

Lista pods

kubectl logs pod-name

Muestra logs

kubectl scale deployment/api --replicas=5

Escala pods

Buenas prácticas para contenedores

Imágenes

  • Usa imágenes base oficiales y ligeras (alpine/slim)
  • Minimiza el número de capas
  • Usa .dockerignore para excluir archivos innecesarios
  • Etiqueta las imágenes con versiones específicas
  • Escanea las imágenes en busca de vulnerabilidades

Seguridad

  • No ejecutes contenedores como root
  • Usa secretos para información sensible
  • Limita los recursos (CPU, memoria)
  • Implementa políticas de seguridad de pods
  • Actualiza regularmente las imágenes base

Operaciones

  • Implementa health checks
  • Configura logs estructurados
  • Usa volúmenes para datos persistentes
  • Implementa estrategias de despliegue seguras
  • Monitorea el rendimiento y uso de recursos

Estructura de JWT

Un JWT consta de tres partes: Header, Payload y Signature, separadas por puntos. Cada parte se codifica en Base64.

Claims comunes:
  • iss (issuer): Emisor del token
  • sub (subject): Sujeto del token (generalmente el ID del usuario)
  • exp (expiration time): Tiempo de expiración
  • iat (issued at): Tiempo de emisión
  • aud (audience): Destinatarios del token
